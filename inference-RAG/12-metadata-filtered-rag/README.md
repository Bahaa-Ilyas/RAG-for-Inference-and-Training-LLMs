# Building Metadata-filtered RAG / auto-retrieval From Scratch

This folder is presented as a cookbook-style, runnable tutorial inspired by the organization of [`daveebbelaar/ai-cookbook` hybrid retrieval](https://github.com/daveebbelaar/ai-cookbook/tree/main/knowledge/hybrid-retrieval): local data, local docs, local utils, numbered walkthrough scripts, and a README that explains why each stage exists.

Metadata-filtered RAG converts user constraints into structured filters and applies them before or during retrieval.

## Setup

No package installation or API key is required for the local teaching path.

```bash
python 1-explore-data.py
python 2-build-index.py
python 3-retrieve.py
python 4-run-method.py --query "Where does vendor onboarding require security review?"
python 5-evaluate.py
```

You can also run the examples entrypoint:

```bash
python examples/run_example.py
```

## Why each piece earns its place

- Data exploration catches parser and metadata problems before retrieval tuning starts.
- A local index makes BM25, dense-style scoring, and fusion inspectable.
- A separate retrieval script shows the baseline before the method-specific logic is added.
- The method runner isolates the specific idea for this RAG method.
- The evaluation script gives a small feedback loop instead of judging by vibes.

## Apply this to your own project

Replace the fixtures in `examples/` and rows in `data/corpus.jsonl` with your real PDFs, scanned OCR output, webpages, tables, and tool results. Keep stable IDs, source type, page or URL metadata, content hashes, versions, and access-control fields. Then swap the local utilities for production components one at a time: PDF parser, crawler, BM25 engine, vector store, reranker, verifier, and evaluation harness.

## Method reference

# Metadata-filtered RAG / auto-retrieval

        ## Summary
        Metadata-filtered RAG converts user constraints into structured filters and applies them before or during retrieval.

        ## Problem it solves
        Semantic similarity alone ignores permissions, dates, departments, versions, document types, products, and jurisdiction constraints.

        ## When to use this method
        Use it for enterprise ACLs, versioned docs, regulated content, customer-specific answers, and structured corpora with reliable metadata.

        ## When not to use this method
        Avoid it when metadata is missing, inconsistent, or inferred unreliably; bad filters silently remove the right evidence.

        ## Core components
        Metadata schema, filter extractor, retriever with filter support, ACL/security trimmer, query planner, generator.

        ## Input and output
        Input is a query plus user identity and context. Output is filtered evidence and an answer constrained to authorized, relevant records.

        ## PDF use
        PDF ingestion should store document ID, title, page, section, author, effective date, table flag, OCR confidence, and access-control fields. For text PDFs, preserve page numbers, section headings, captions, reading order, and chunk offsets. For scanned PDFs, run OCR before chunking and store OCR confidence, page image references, and detected tables. Use Unstructured, Docling, Marker, PyMuPDF, pdfplumber, Apache Tika, or Tesseract depending on document quality. Long scientific papers usually need section-aware chunking, parent-child links, equation/caption retention, and table extraction.

        ## Webpage use
        Web metadata should include canonical URL, domain, crawl timestamp, content hash, language, version, and freshness policy. For webpages, crawl with robots.txt awareness, sitemap support, canonical URL normalization, crawl timestamps, content hashing, boilerplate removal, and duplicate detection. Use Trafilatura, BeautifulSoup, Scrapy, Playwright, readability extraction, and URL canonicalization. Treat retrieved web text as untrusted because prompt injection can be embedded in pages.

        ## Production stack
        LlamaIndex auto-retrieval, Weaviate/Qdrant/Milvus filters, OpenSearch filters, Azure AI Search filters, LangChain self-query retriever.

        ## Minimal example
        A query 'latest EU policy for vendor onboarding' becomes filters region=EU, document_type=policy, effective_date<=today, superseded=false.

        ## Pros
        - Prevents wrong-scope answers
- Essential for permissions and versions
- Reduces irrelevant candidates

        ## Cons
        - Requires disciplined metadata governance
- Filter extraction can be wrong
- Over-filtering hurts recall
